1. Contexto e Problema

Atualmente o Severus Barber opera em modo offline (SQLite local). Cada dispositivo tem sua própria base de dados isolada, sem sincronização. Isso impede que:
O dono (Admin) veja em tempo real o que os barbeiros estão cadastrando e vendendo.
Um barbeiro instalado em seu próprio celular acesse os clientes cadastrados por outro.
O Admin crie uma conta de login para um novo barbeiro diretamente pelo app.
Comandas, produtos e financeiro fiquem centralizados em um único banco compartilhado.

O objetivo deste PRD é especificar todas as modificações necessárias para que o Firebase (Auth + Firestore) seja ativado como fonte principal de dados, mantendo o SQLite apenas como cache offline.

2. Arquitetura Alvo

A arquitetura deve seguir o modelo abaixo após a integração:

  [Celular do Admin]  ←──── Firestore (nuvem) ────→  [Celular do Barbeiro 1]
  [Celular do Admin]  ←──── Firestore (nuvem) ────→  [Celular do Barbeiro 2]

Regras gerais:
Firebase Auth gerencia a identidade de todos os usuários (Admin + Barbeiros).
Cloud Firestore é o banco principal — todos os writes e reads passam por ele quando há internet.
SQLite continua existindo como cache local para modo offline — ao reconectar, sincroniza com Firestore.
Cada documento no Firestore tem o campo barbearia_id (ou shop_id) para isolar dados de barbearias diferentes no futuro (arquitetura SaaS multi-tenant).
2.1 Coleções no Firestore
Estrutura de coleções a ser criada/validada:

barbearias/{barbearia_id}/
    usuarios/{uid}          → Admin e Barbeiros
    clientes/{id}           → Clientes cadastrados
    servicos/{id}           → Serviços oferecidos
    produtos/{id}           → Produtos para venda
    comandas/{id}           → Comandas abertas e fechadas
    comandas/{id}/itens/{id}→ Itens de cada comanda
    atendimentos/{id}       → Histórico de atendimentos
    despesas/{id}           → Despesas financeiras
    agendamentos/{id}       → Agenda
    caixas/{id}             → Abertura/fechamento de caixa
    estoque/{id}            → Movimentações de estoque

Cada documento deve conter os campos: created_at, updated_at, created_by (uid do usuário que criou).

3. Feature A — Firebase Auth: Admin Cria Login do Barbeiro

3.1 Problema Atual
O auth_service.dart já importa Firebase Auth, mas a criação de conta de barbeiro provavelmente usa createUserWithEmailAndPassword de forma que desloga o Admin (comportamento padrão do Firebase no cliente). Isso quebra a sessão do dono.
3.2 Solução Requerida
Usar uma segunda instância do Firebase App para criar a conta do barbeiro sem afetar a sessão atual do Admin. Segue o padrão exato a ser implementado:

// auth_service.dart
Future<UserCredential> criarContaBarbeiro({
  required String email,
  required String senha,
  required String nome,
  required String telefone,
  required double comissaoPercentual,
}) async {
  // 1. Cria segunda instância Firebase para não deslogar o Admin
  final secondaryApp = await Firebase.initializeApp(
    name: 'secondaryApp_${DateTime.now().millisecondsSinceEpoch}',
    options: Firebase.app().options,
  );
  final secondaryAuth = FirebaseAuth.instanceFor(app: secondaryApp);

  // 2. Cria a conta do barbeiro
  final credential = await secondaryAuth.createUserWithEmailAndPassword(
    email: email, password: senha,
  );

  // 3. Salva dados no Firestore
  await FirebaseFirestore.instance
    .collection('barbearias')
    .doc(barbearia_id)
    .collection('usuarios')
    .doc(credential.user!.uid)
    .set({
      'uid': credential.user!.uid,
      'nome': nome,
      'email': email,
      'telefone': telefone,
      'role': 'barbeiro',
      'comissao_percentual': comissaoPercentual,
      'first_login': true,
      'ativo': true,
      'created_at': FieldValue.serverTimestamp(),
    });

  // 4. Deleta a instância secundária
  await secondaryApp.delete();
  return credential;
}

4. Feature B — Sincronização Firestore de Todas as Entidades

Cada service existente deve ser atualizado para ler e escrever no Firestore como fonte primária. O SQLite deve ser mantido como cache (mirror local). A lógica padrão de cada service:

// Padrão de escrita (write-through cache)
Future<void> salvar(Entidade e) async {
  await _firestore.collection(...).doc(e.id).set(e.toMap()); // Firestore primeiro
  await _db.insert(..., e.toMap());                          // SQLite como cache
}

// Padrão de leitura (online: Firestore | offline: SQLite)
Stream<List<Entidade>> listar() {
  if (conectado) return _firestore.collection(...).snapshots()...;
  return Stream.value(await _db.query(...));
}
4.1 cliente_service.dart
Criar, editar e deletar clientes deve refletir em Firestore em tempo real.
Campos adicionais obrigatórios no documento: barbearia_id, created_by, data_nascimento.
Usar StreamBuilder ou Provider com stream do Firestore na tela de listagem de clientes.
Barbeiro vê todos os clientes da barbearia (não só os dele).
4.2 produto_service.dart
Produtos são cadastrados pelo Admin — barbeiros apenas consultam e usam em comandas.
Ao baixar estoque (fechar comanda), atualizar o campo quantidade no Firestore.
Alertas de estoque mínimo devem ser baseados no valor do Firestore.
4.3 servico_service.dart
Serviços cadastrados pelo Admin e visíveis para todos os barbeiros.
Sincronizar campo comissao_percentual — mas lembrar que a Feature 1 (PRD anterior) usa o percentual do barbeiro, não do serviço.
4.4 comanda_service.dart
Comanda aberta: documento com status 'aberta' no Firestore — Admin consegue ver em tempo real quais comandas estão abertas.
Ao fechar comanda: atualizar status para 'fechada', registrar comissão calculada, dar baixa no estoque.
Campo barbeiro_uid na comanda identifica qual barbeiro realizou o atendimento.
Admin consegue ver TODAS as comandas. Barbeiro vê apenas as suas.
4.5 financeiro_service.dart
Despesas cadastradas pelo Admin salvas no Firestore.
Dashboard do Admin lê faturamento consolidado de todos os barbeiros via Firestore.
Lucro líquido = soma de todas as comandas fechadas - despesas do período.
4.6 agenda_service.dart
Agendamentos salvos no Firestore — barbeiro vê sua própria agenda, Admin vê todas.
Usar Timestamp do Firestore para data/hora dos agendamentos.
4.7 estoque_service.dart (movimentos)
Toda movimentação de estoque (entrada manual, baixa por comanda) salva no Firestore com quem fez, quando e qual produto.

5. Regras de Segurança do Firestore

Implementar as seguintes Firestore Security Rules no console do Firebase:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function isAuth() { return request.auth != null; }

    function getUser() {
      return get(/databases/$(database)/documents/
        barbearias/$(request.auth.token.barbearia_id)/
        usuarios/$(request.auth.uid)).data;
    }

    function isAdmin() { return getUser().role == 'admin'; }
    function isBarbeiro() { return getUser().role == 'barbeiro'; }

    // Usuários: Admin lê e escreve todos; Barbeiro lê apenas o próprio
    match /barbearias/{shopId}/usuarios/{uid} {
      allow read: if isAuth();
      allow write: if isAuth() && isAdmin();
    }

    // Clientes, Serviços, Produtos: todos leem; só Admin escreve
    match /barbearias/{shopId}/clientes/{id} {
      allow read: if isAuth();
      allow write: if isAuth() && (isAdmin() || isBarbeiro());
    }

    match /barbearias/{shopId}/servicos/{id} {
      allow read: if isAuth();
      allow write: if isAuth() && isAdmin();
    }

    match /barbearias/{shopId}/produtos/{id} {
      allow read: if isAuth();
      allow write: if isAuth() && (isAdmin() || isBarbeiro());
    }

    // Comandas: barbeiro cria/edita as suas; Admin lê todas
    match /barbearias/{shopId}/comandas/{id} {
      allow read: if isAuth();
      allow create: if isAuth() && (isAdmin() || isBarbeiro());
      allow update: if isAuth() &&
        (isAdmin() || resource.data.barbeiro_uid == request.auth.uid);
      allow delete: if isAuth() && isAdmin();
    }

    // Financeiro: somente Admin
    match /barbearias/{shopId}/despesas/{id} {
      allow read, write: if isAuth() && isAdmin();
    }

    // Demais coleções: autenticados leem e escrevem
    match /barbearias/{shopId}/{collection}/{id} {
      allow read, write: if isAuth();
    }
  }
}

6. Fluxo Completo de Uso (do Zero)

6.1 Configuração Inicial (uma única vez)
Passo 1 — Admin instala o APK no seu celular.
Passo 2 — Na tela de login, toca em 'Criar conta Admin' (visível apenas se não houver Admin cadastrado no Firestore).
Passo 3 — Preenche nome, email e senha. O app cria o documento na coleção usuarios/ com role = 'admin' e barbearia_id gerado automaticamente.
Passo 4 — Admin loga e acessa o Dashboard Administrativo.
6.2 Adicionando um Barbeiro
Admin vai em Menu → Barbeiros → Adicionar Barbeiro.
Preenche nome, email, senha temporária e % de comissão.
App cria a conta via segunda instância Firebase (sem deslogar o Admin).
Documento do barbeiro salvo no Firestore com first_login = true.
Admin comunica o email e senha temporária ao barbeiro (WhatsApp, etc).
6.3 Barbeiro Instalando o App
Barbeiro baixa o APK e instala no próprio celular.
Abre o app — tela de Login.
Digita email e senha temporária recebidos do Admin.
App detecta first_login = true → redireciona para tela 'Crie sua nova senha'.
Após redefinir senha → acessa o Dashboard do Barbeiro com todos os dados da barbearia já sincronizados.
6.4 Barbeiro no Dia a Dia
Abre comanda → seleciona cliente (lista vem do Firestore, compartilhada).
Adiciona serviços e produtos — comissão calculada com o % do barbeiro.
Fecha comanda → dados gravados no Firestore → Admin vê em tempo real no próprio celular.
Cadastra novo cliente → aparece para todos os barbeiros instantaneamente.

9. Restrições e Premissas

Não remover o SQLite — mantê-lo como cache offline obrigatório.
Não quebrar nenhuma funcionalidade existente (comandas, estoque, financeiro, agenda).
Todo texto para o usuário em pt-BR.
Seguir padrão de cores: #D4AF37 (amarelo mecânico), #1A1A1A (preto), #FFFFFF.
Manter arquitetura: UI → Controller → Service → Firestore/SQLite.
Não fazer migrations destrutivas no SQLite — usar ALTER TABLE com ADD COLUMN.
Testar em dispositivo Android real (não apenas emulador) para validar Firebase Auth.
